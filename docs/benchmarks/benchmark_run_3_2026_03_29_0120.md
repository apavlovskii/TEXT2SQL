## What We Built

We built an agentic Text-to-SQL system targeting the Spider2-Snow benchmark.

- Retrieves a compact, relevant subset of database schema (SchemaSlice) using a vector database, instead of sending full schemas to the LLM.
- Generates SQL via a structured plan-first approach to improve correctness and reduce hallucinations.
- Executes queries against Snowflake and automatically classifies/repairs errors through a bounded refinement loop.
- Generates multiple candidate SQL queries (Best-of-N) to increase the chance of finding a correct solution.
- Applies semantic verification using result fingerprints, expected output shape, and consistency checks.
- Stores successful query traces and reuses them through a memory retrieval mechanism.
- Uses a join graph to improve schema connectivity and multi-table query reasoning.
- Uses a learned verifier to improve candidate selection from past execution signals.

### Architecture updates since Benchmark Run 2

The following improvements were implemented based on the error analysis from Run 2 (where case sensitivity and VARIANT access were the dominant remaining issues):

- **Double-quoted column identifiers everywhere:** The SQL compiler now wraps every column reference in double-quotes (`"fullVisitorId"`, `"publication_number"`) to preserve Snowflake's case-sensitive identifier resolution. This addresses 63% of the "invalid identifier" errors from Run 2.
- **Original column casing preserved end-to-end:** `ColumnSlice` now stores `original_name` (exact case from Snowflake's `DESCRIBE TABLE`). The compiler looks up original casing via a case map and emits `"fullVisitorId"` instead of `"FULLVISITORID"`.
- **VARIANT column awareness in prompts:** VARIANT columns are flagged in the SchemaSlice with `is_variant=True` and the prompt text shows "(VARIANT -- use :field path)" so the LLM knows to use colon access syntax (`"trafficSource":"source"::STRING`).
- **Early termination on hopeless repairs:** If the same error type (e.g., `invalid_identifier`) repeats 3+ times across repair attempts, the loop breaks early instead of burning remaining budget. Saves ~30% of wasted tokens.
- **LATERAL FLATTEN guidance in prompts:** The Snowflake guidance section now includes explicit LATERAL FLATTEN examples and double-quoting rules. Repair prompts also instruct the LLM to always quote column names.
- **Trace memory SQL preview:** Retrieved memory traces now include truncated SQL snippets so the LLM can see exact column quoting and access patterns from prior successful queries.
- **Snowflake syntax reference during repairs:** When a repair is triggered, the system queries the `snowflake_syntax` ChromaDB collection for relevant SQL syntax documentation and injects it into the repair prompt (e.g., FLATTEN syntax for VARIANT errors, JOIN syntax for join errors).
- **Snowflake syntax reference collection:** 55 chunks across 11 topics (JOIN, LATERAL FLATTEN, CTEs, QUALIFY, GROUP BY, WHERE, ORDER BY, PIVOT, set operators, subqueries, identifier quoting rules) ingested into a new `snowflake_syntax` ChromaDB collection using `text-embedding-3-large`.
