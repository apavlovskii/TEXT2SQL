# Benchmark Run 9

## What We Built

Same core agent as Run 8, with significant improvements to data profiling, schema representation, SQL generation reliability, and candidate scoring.

### Architecture updates since Benchmark Run 8

1. **Data profiling pipeline** (`scripts/profile_data.py`) — Extracts 100 rows per table from Snowflake (deduplicating partitioned tables), profiles each column (null rates, unique counts, value ranges, VARIANT structure), then uses GPT-5.4 to generate natural language descriptions for every table and column. Descriptions are stored in `data/table_column_descriptions.json` and enriched into ChromaDB `schema_cards`.

2. **GPT-5.4 generated descriptions** — All 9 tables and 204 columns across GA360, GA4, PATENTS, PATENTS_GOOGLE now have LLM-generated descriptions based on actual data profiles. These descriptions are embedded into ChromaDB document text, improving semantic search quality and providing the LLM with precise column semantics, VARIANT access patterns, and value format guidance.

3. **Universal Pydantic type coercion** — Replaced per-field `field_validator` with `_CoercingBase` base class using `model_validator(mode="before")`. All plan models (PlanJoin, PlanFilter, PlanAggregation, PlanOrderBy, PlanFlatten, PlanCTE, QueryPlan) now automatically coerce integer/boolean values to strings, eliminating plan parse failures from LLM-generated JSON.

4. **GA360/GA4 partition table collapsing** — Merged 366 GA360 daily tables into 1 representative and 92 GA4 daily tables into 1 representative in ChromaDB. Groups by schema+base_name (ignoring column differences), takes union of all columns, keeps the latest partition name (which exists in Snowflake). Partition comment instructs the LLM to filter by the `"date"` column.

5. **VARIANT sub-field enrichment** — `_enrich_variant_fields()` in `debug_retrieve.py` queries ChromaDB for VARIANT_FIELD entries, groups by parent column, and attaches field names to `ColumnSlice.variant_fields`. Uses the VARIANT_FIELD comment to distinguish ARRAY (needs FLATTEN) from OBJECT (colon access). VARIANT_FIELD sub-columns are filtered out of the schema prompt (their info is carried by the parent).

6. **FLATTEN+OBJECT_KEYS fallback** — For VARIANT columns declared as `VARIANT` (not `ARRAY`), the indexer now tries `LATERAL FLATTEN + OBJECT_KEYS` as a fallback when `OBJECT_KEYS` alone fails. This discovered 93 sub-fields for PATENTS (previously 0).

7. **Revenue ÷10^6 hint** — Added to `_SNOWFLAKE_GUIDANCE` in prompt_builder.py: "GA360 revenue fields (totalTransactionRevenue, productRevenue, transactionRevenue) are stored multiplied by 10^6. ALWAYS divide by 1000000 to get USD values."

8. **Candidate scoring fix** — `execution_success` in `_candidate_to_result` is now True when SQL executed and returned rows, even if gold verification failed. Previously, gold-mismatch candidates scored 0.0 (same as errors); now they score 100+ (execution success bonus + shape bonuses). This prevents the selector from defaulting to `SELECT 1` when real SQL candidates exist.

9. **Plan retry + LLM SQL fallback** — In `candidate_generator.py`, when `compile_plan()` produces `SELECT 1` (empty `selected_tables`), the system now: (a) retries with feedback telling the LLM to include tables, then (b) falls back to direct LLM SQL generation bypassing the deterministic compiler.

10. **Increased retrieval budget** — `config/defaults.yaml` updated: `max_schema_tokens: 10000` (was 2500), `top_k_columns: 100` (was 25), `top_k_tables: 25` (was 8). This ensures all critical columns (hits, totals, date, fullVisitorId) are included in the GA360 schema prompt.

11. **Comment wiring in schema slice** — `build_schema_slice` in `debug_retrieve.py` now passes `comment` from ChromaDB metadata to `ColumnSlice` and `TableSlice`, so LLM-generated descriptions flow through to `format_for_prompt()`.

12. **Gold verification path fix** — Corrected `--gold_dir` from `.../gold/sql/` to `.../gold/`. Run 8 was the first to verify against actual gold execution output. This run continues with correct gold verification.

## Benchmark Parameters

- Generate 8 candidate queries on each iteration
- Use GPT-5.4 model
- ChromaDB with GPT-5.4 profiled descriptions
- Gold verification against execution output
