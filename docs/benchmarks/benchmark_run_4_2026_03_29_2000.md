### Architecture updates since Benchmark Run 3

The following improvement was implemented based on the finding that Run 3's 92% execution accuracy only translated to 8.7% gold-match accuracy — meaning queries compiled and ran but returned wrong results:

- **Gold-match verification in the retry loop:** After SQL executes successfully, the system now compares the actual query results against gold CSV files from Spider2's evaluation suite. If the results don't match, the execution is treated as a **failure** and enters the repair loop with a specialized prompt: "SQL executed but returned WRONG RESULTS — re-read the question, check filters/joins/aggregations/date conditions." This replaces the previous behavior where any successful execution was accepted as correct.
- **New error types:** `result_mismatch` (query returned wrong data) and `empty_result` (query returned no rows when results were expected) added to the error taxonomy. These are tracked in repair traces and final reports alongside compilation and identifier errors.
- **Result-aware repair prompts:** When a gold mismatch is detected, the repair prompt explicitly tells the LLM that the SQL ran successfully but produced incorrect output, and asks it to re-examine the question's conditions, joins, aggregations, and date/time filters. This is qualitatively different from identifier/syntax repair prompts.
- **Early termination on repeated mismatches:** If gold mismatch occurs 3+ times for the same candidate, the system stops retrying and moves to the next candidate, avoiding token waste on semantically broken queries.
- **`--gold_dir` CLI flag:** The experiment runner accepts `--gold_dir Spider2/spider2-snow/evaluation_suite/gold` to enable gold-match verification. Without this flag, behavior is unchanged (execution-only accuracy). Results now include a `gold_matched` field (true/false/null).
- **Gold comparison logic:** Uses the exact same comparison algorithm as Spider2's `evaluate.py` — tolerance of 1e-2 for numeric comparisons, NaN normalization, optional `ignore_order`, `condition_cols` support, and multi-gold-file matching.
