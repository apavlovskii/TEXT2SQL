## What We Built

We built an agentic Text-to-SQL system targeting the Spider2-Snow benchmark.

- Retrieves a compact, relevant subset of database schema (SchemaSlice) using a vector database, instead of sending full schemas to the LLM.
- Generates SQL via a structured plan-first approach to improve correctness and reduce hallucinations.
- Executes queries against Snowflake and automatically classifies/repairs errors through a bounded refinement loop.
- Generates multiple candidate SQL queries (Best-of-N) to increase the chance of finding a correct solution.
- Applies semantic verification using result fingerprints, expected output shape, and consistency checks.
- Stores successful query traces and reuses them through a memory retrieval mechanism.
- Uses a join graph to improve schema connectivity and multi-table query reasoning.
- Uses a learned verifier to improve candidate selection from past execution signals.

### Architecture updates since Benchmark Run 1

The following improvements were implemented based on the error analysis from Run 1 (where 106 "invalid identifier" errors were the dominant failure mode):

- **VARIANT sub-field indexing:** Snowflake VARIANT/OBJECT/ARRAY columns (e.g., `trafficSource`, `hits`, `totals` in GA360/GA4) now have their nested field paths extracted via `OBJECT_KEYS()` sampling and indexed as separate ColumnCards (e.g., `trafficSource:source`, `device:browser`, `hits:page`). This added 58 sub-field columns across GA4 (+21) and GA360 (+37).
- **Pre-execution column validation:** Before submitting SQL to Snowflake, all column references are now verified against the ChromaDB index. Invalid columns are flagged with suggested alternatives, and a targeted repair prompt fixes them before execution — saving a Snowflake round-trip.
- **Iterative retrieval (plan-guided schema expansion):** After the LLM generates a draft plan, any tables or columns referenced in the plan but missing from the SchemaSlice are automatically fetched from ChromaDB and added (capped at 3 extra tables, 1 expansion round).
- **Micro-probes for uncertainty:** When column validation is uncertain, up to 2 live Snowflake probes (`SELECT "col" FROM table LIMIT 0`) confirm whether a column actually exists, enriching repair prompts with confirmed-missing information.
- **Gold SQL join extraction:** 234 high-confidence JoinCards were extracted from 120 gold SQL files in `Spider2/spider2-snow/evaluation_suite/gold/sql/` and ingested into ChromaDB (`source="gold_sql"`, `confidence=1.0`), providing ground-truth join paths for relation-aware retrieval.
- **Embeddings upgraded to `text-embedding-3-large`** (OpenAI, 3072 dimensions) for stronger semantic matching during schema retrieval.

Overall goal: achieve high accuracy on Spider2 while significantly reducing token usage and improving efficiency compared to existing approaches like ReFoRCE.
